# 코드 스타일

## 컴포넌트 및 함수

- 컴포넌트 정의 방식: 화살표 함수 vs 함수 선언
  - 컴포넌트(함수)의 자체 선언은 함수 선언식을 사용한다.
  - 함수 내부에서는 화살표 함수를 사용한다.
    - 이벤트 핸들러 등
- export 방식: default export vs named export.
  - Nextjs는 page, layout 등은 default export가 강제
    - 이 때 페이지를 제외한 나머지들은 named로 강제한다.
- Props 타입 위치와 작성 방식.
  - type vs interface 중 interface를 기본으로 사용한다.
  - props spreading
    - 금지하면 컴포넌트가 어떤 값을 사용해서 넘기는지 잘 알 수 있다.
    - 반대로 html 태그는 props가 많아서 일일히 입력하기 매우 어렵다.
    - 따라서 컴포넌트는 금지, html 태그는 허용한다.

## 파일 포맷

- 파일 길이 제한: 30자
- 함수 길이 제한: 파일 길이와 동일하게
- 줄 길이 제한: 80줄
- 줄바꿈 설정은 전부 window 환경이므로 그대로 CRLF 사용한다.
  - 단, `.husky/*` 스크립트는 유닉스 셸에서 실행되므로 예외적으로 LF를 사용한다.
- 줄 끝 공백은 없앤다.
- 들여쓰기 칸수는 2칸이다.
- 따옴표는 큰따옴표를 사용한다.
- 객체 내부의 값 맨 끝에 쉼표를 사용한다.
- 들여쓰기 시 Tab을 사용하지 않고 스페이스바를 사용한다.

## import

- import 정렬과 alias 사용 규칙.
  - import
  - 파일을 `import`할 때 확장자(`.js`, `.tsx` 등)를 붙이지 않도록 강제한다.
  - `@/components`, `@/hooks`와 같은 alias를 사용한다.
    - 단, 상대 경로를 완전히 막지는 않는다.

## type

- `any`/`unknown` 은 허용하지 않는다.

## 확장자

- JSX(React 컴포넌트) 문법은 오직 `.tsx`와 `.jsx` 확장자를 가진 파일에서만 작성할 수 있다.

    - 언제 이러한 연산자들을 사용할지?
      - 개인적 의견: 간단한 경우 (bool && )의 경우에만 사용
        - 이외의 경우는 별도 함수(컴포넌트)로 분리
